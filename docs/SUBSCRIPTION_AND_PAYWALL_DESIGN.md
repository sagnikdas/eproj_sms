# Subscription, Paywall & Promo Codes — Design Plan

**Audience:** Caregivers / children (payers) who subscribe and advocate installing the app on their parent’s phone.  
**Goal:** Define wrapper screens, paywall placement, payment flow, and promo/discount codes.

---

## 1. Actor model

| Actor | Role | Where they interact |
|-------|------|----------------------|
| **Caregiver (payer)** | Pays subscription; gets “activation” for parent’s device | Web portal **or** same app during “I’m setting up for my parent” flow |
| **Parent (end user)** | Uses the app on their phone | This app (Elder Shield) on their device |

Payment is always tied to the **caregiver**. The **parent’s device** only needs to be “activated” (linked to a paid subscription or a valid code).

---

## 2. How we know the mapping (caregiver ↔ parent)

We need a clear link between **who paid** (caregiver) and **whose device is protected** (parent). That mapping is stored on the **backend**; the app only sends the activation code (and optionally a device/parent identifier).

### 2.1 What we need to store

| Concept | Who provides it | Where stored |
|--------|------------------|--------------|
| **Payer (caregiver)** | Email/phone at payment (web) | Backend: subscription or “license” record |
| **Activation code** | Generated by backend after payment/promo | Backend: linked to one subscription |
| **Parent / device** | When code is used in the app | Backend: “this code was used on device X” (and optionally parent phone/name) |

So the mapping is: **caregiver (payer) → subscription → activation code → device (parent’s phone)**. Optionally we add an explicit “parent” label (name or phone) for your own visibility (support, analytics, “devices under this caregiver”).

### 2.2 Where we capture caregiver identity

- **At payment (web paywall):** Caregiver enters email (and optionally phone). Payment provider gives you payer email. Backend creates:
  - A **subscription** (or “license”) tied to that email.
  - One **activation code** (one code = one parent’s device).

So we **always** know the caregiver from payment: they are the subscription owner (email/phone from checkout).

### 2.3 Where we capture parent / device identity

Choose one (or combine):

**Option A — Device-only (minimal)**  
- When the app calls `POST /validate-activation` with the code, send an **opaque device id** (or install id) from the app.  
- Backend stores: `subscription_id → activation_code → device_id` (and marks code as “used”).  
- Mapping: we know **payer email** (from subscription) and **device id** (from first activation). We do **not** know parent name or phone unless we ask.

**Option B — Parent identifier at activation (recommended)**  
- On **ActivationScreen** in the app, optionally ask: “Parent’s phone number” (or name).  
- App sends: `{ "code": "ES-XXXX", "parentPhone": "+91..." }` (or `parentName`).  
- Backend stores: subscription → code → device_id + parent_phone (or name).  
- Then you have: **caregiver email ↔ parent phone** (or name). Useful for support (“which device is Mom’s?”) and for future features (e.g. SMS reminder to parent).

**Option C — Parent identifier at payment**  
- On **web paywall**, caregiver enters “This subscription is for: [Parent’s phone or name].”  
- Backend stores: subscription (caregiver email) + parent_phone (or name). When code is used, you link device_id to that subscription, so you already have caregiver ↔ parent from payment.  
- Activation in the app can stay code-only (no need to ask parent phone again).

### 2.4 One caregiver, multiple parents

- One caregiver may pay for **multiple** parents (e.g. Mom and Dad, or parent + in-laws).  
- Model: **one subscription per parent** (or “per device”), or one subscription with **multiple licenses/codes**.  
  - Simple: each payment (or “Add another parent” on web) creates a **new subscription** and a **new activation code**.  
  - Then: caregiver_email ↔ many subscriptions ↔ many codes ↔ many devices.  
- Backend: table `subscriptions` (id, payer_email, parent_label?, created_at) and `activations` (subscription_id, code, device_id?, parent_phone?, used_at).

### 2.5 One parent, multiple caregivers

- Less common for payment, but possible (e.g. two siblings share cost).  
- You can either:  
  - Allow **one active subscription per device** (first code that activates “wins”), or  
  - Support **co-payers** in backend (same subscription_id, multiple payer emails).  
- For MVP, “one device = one subscription = one payer” is enough; you can add co-payer later.

### 2.6 Summary: how we know the mapping

| We know | How |
|--------|-----|
| **Caregiver** | From payment (web): payer email/phone on subscription. |
| **Parent’s device** | From first activation: app sends code + device_id (and optionally parent_phone/name). |
| **Caregiver ↔ Parent** | Backend links subscription (payer) ↔ activation (device + optional parent_phone). Optionally caregiver or parent provides the other’s label at payment or activation. |

So: **customers (caregiver or parent) mapping** is “subscription owner (caregiver) + activation record (device/parent).” Store it in backend tables (e.g. `subscriptions`, `activations`); the app only needs to send code + optional parent identifier when activating.

## 3. High-level flows

### 3.1 Caregiver pays → Parent activates (recommended)

1. **Caregiver** (on their phone or laptop):
   - Visits **web paywall** (your site or payment provider’s hosted page).
   - Subscribes (monthly/yearly) or applies **promo code** (free / discount).
   - After payment (or free code): receives an **activation code** (e.g. `ES-XXXX-XXXX`) or **activation link** (e.g. `elder shield://activate?token=...` or `https://yourapp.com/activate?token=...`).

2. **Parent’s device** (this Flutter app):
   - Caregiver installs app, opens it.
   - App shows **Activation** screen (pre or post onboarding): “Enter the code from the person who set up your subscription.”
   - User (or caregiver) enters code (or opens activation link).
   - Backend validates code → returns subscription status + optional expiry.
   - App stores “activated” state locally (and optionally refresh token).
   - App unlocks and shows normal flow (onboarding if first time → then main app).

So: **paywall + payment + promo codes live on the web (or a separate caregiver app)**; **this app only does “activation” and subscription state**.

### 3.2 Alternative: Paywall inside this app

If the **same person** installing the app is the payer (e.g. caregiver at parent’s home):

1. First launch → **Pre-onboarding paywall** (“Subscribe to protect your parent’s phone”).
2. User pays (in-app purchase or redirect to payment gateway).
3. Optional: **Promo code** field on paywall (free / discount).
4. On success → continue to existing onboarding (Welcome → Permissions → Trusted contact) → main app.

Here the “wrapper” is: **Paywall → Onboarding → LaunchGate → MainShell**.

---

## 4. Where to put the “wrapper” in this codebase

Current flow in `app.dart`:

```
_onboardingComplete == null  → loading
_onboardingComplete == false → OnboardingFlow → then set true → LaunchGate
_onboardingComplete == true  → LaunchGate → MainShell
```

Two clean insertion points:

### Option A — Activation before onboarding (recommended if activation code is primary)

- New state: **“activated”** (subscription or valid code).
- Order: **Activation gate → Onboarding → LaunchGate → MainShell**.

```
_onboardingComplete == null  → loading
_activated == false           → ActivationScreen (enter code / open link)
_activated == true && !onboarding → OnboardingFlow
else                          → LaunchGate → MainShell
```

- **ActivationScreen:** Input for code (and optionally deep link handling). Call backend to validate; persist `activated` + optional expiry in `SettingsService` / secure storage.

### Option B — Activation after onboarding

- Same **ActivationScreen**, but shown **after** onboarding and before **LaunchGate**.
- User can complete onboarding (Welcome, Permissions, Trusted contact), then sees “Enter activation code from your caregiver.”
- Pros: Parent sees product first. Cons: More steps before “fully usable.”

### Option C — Paywall inside this app (no separate web)

- **PaywallScreen** before onboarding (e.g. “Subscribe or enter promo code”).
- Payment gateway: open in WebView or external browser; on success redirect back with token (or poll backend).
- Then **OnboardingFlow** → **LaunchGate** → **MainShell**.

Recommended for your “caregiver pays, parent uses” model: **Option A** (activation gate first), with payment and promo codes handled on a **web paywall**, and this app only doing activation + subscription state.

---

## 5. Paywall / payment gateway (caregiver side)

- **Where:** Web app or hosted payment page (e.g. Stripe Checkout, Paddle, or your own page with Stripe/Paddle/Razorpay).
- **Inputs:** Plan (monthly/yearly), **promo code** (optional).
- **Outputs:** Payment success → generate **activation code** (or link) and send to caregiver (email / show on screen).
- **Backend:** Store subscription (e.g. by email/phone), map `activation_code → subscription_id`, and expose an API for the app: “validate code” and “get subscription status.”

No paywall inside the Flutter app is strictly required if you use a web paywall; the app only needs to **call the “validate activation code” API** and store the result.

## 6. Promo / “cheat” codes

| Type | Purpose | Where validated |
|------|--------|-----------------|
| **Free giveaway** | 100% discount; no payment | Same backend as payment: code → grant subscription (e.g. 1 year) and return activation code or “already activated” token. |
| **Discount** | X% off or fixed amount off | Paywall: apply discount before charging; then same flow as paid (activation code after “payment” of 0 or reduced amount). |
| **Trial** | e.g. 14 days free then pay | Backend marks subscription as trial; app shows “active until &lt;date&gt;”; paywall can show “subscribe to continue” when trial ends. |

Design:

- **Code format:** e.g. `ELDERSHIELD-FREE-2025` or `ES50-MARCH` (50% off). Stored in your backend with type (full free / discount %) and optional expiry/cap.
- **Where entered:**  
  - **Web paywall:** “Have a promo code?” field before or at checkout.  
  - **App (if you add in-app paywall):** Same field on PaywallScreen.  
  - **App (activation-only):** Optional: “I have a code from my caregiver” could be the same activation code; or a separate “promo” entry that backend resolves to an activation token.
- **Validation:** Always **server-side** (backend). App only sends code to backend and trusts response (active / not found / expired).

## 7. Pre-screens / wrapper summary

| Screen | When shown | Purpose |
|--------|------------|--------|
| **ActivationScreen** | First launch, before or after onboarding | Enter activation code (or handle activation link); call backend; persist `activated` + expiry. |
| **PaywallScreen** (optional, in-app) | If you want in-app payment | Subscribe or enter promo; redirect to payment; on success mark paid and optionally get activation code for this device. |
| **Existing OnboardingFlow** | After activation (or after paywall if in-app) | Welcome → Permissions → Trusted contact. |
| **LaunchGate** | After onboarding | Existing: launch SMS check → full-screen warning or MainShell. |
| **MainShell** | When activated + onboarding done | Home, Messages, Settings. |

Minimal addition for “caregiver pays on web, parent activates in app”:

- **One new pre-screen:** **ActivationScreen** (code entry + deep link).
- **New persisted state:** `activated` (and optionally subscription expiry).
- **New backend:** Validate activation code (and optionally return expiry); create code when payment or free/discount promo is used on web.

## 8. Data and backend (minimal)

- **Storage (app):**
  - `activation_token` or `activation_code` (opaque token to refresh status).
  - `subscription_expiry` (optional; for trials or grace period).
  - `activated` boolean (or derive from token non-null).

- **Backend API (e.g. REST):**
  - `POST /validate-activation`  
    Body: `{ "code": "ES-XXXX-XXXX", "deviceId": "opaque-install-id", "parentPhone": "+91..." }` (deviceId and parentPhone optional but recommended for mapping).  
    Response: `{ "valid": true, "expiresAt": "2026-03-12T00:00:00Z" }` or `{ "valid": false, "reason": "expired" }`.  
    Backend stores activation (subscription_id, code, device_id, parent_phone, used_at) so you have **caregiver ↔ parent/device**.
  - When caregiver pays or uses promo on web: create subscription (payer email) + activation code, link code to subscription; return code (or send by email).

- **Mapping tables (caregiver ↔ parent):**
  - `subscriptions`: id, payer_email, parent_label (optional, from paywall), plan, expires_at, created_at.
  - `activations`: id, subscription_id, code, device_id (from app), parent_phone or parent_name (optional, from app), used_at.
  - So: caregiver = subscriptions.payer_email; parent/device = activations.device_id (+ activations.parent_phone if you collect it).

- **Promo codes:** Stored in backend; applied at payment (web) or in a dedicated “redeem promo” endpoint that returns an activation code.

## 9. Implementation checklist (this repo)

- [ ] Add **subscription/activation** keys to `SettingsService` (e.g. `activationToken`, `subscriptionExpiry`, or `isActivated`).
- [ ] Add **ActivationScreen** (code text field, “Activate” button, optional “Open link” / deep link handling).
- [ ] In **app.dart**: resolve `activated` state; if not activated, show **ActivationScreen** instead of onboarding/main app; on success set `activated` and continue.
- [ ] Optional: **Deep link** (e.g. `elder_shield://activate?code=...`) to open app and prefill or auto-submit code.
- [ ] **Backend** (separate repo or service): paywall page, payment gateway, promo code logic, activation code generation, and `POST /validate-activation`.
- [ ] (Optional) **PaywallScreen** in-app if you want in-app payment and promo entry; integrate payment redirect and token handling.

This keeps the Flutter app focused on **activation and UX**; payment and promo logic stay on the server and web paywall, which fits the “caregiver pays, parent uses” model and keeps cheat codes manageable (create/revoke on backend).

## 10. Phased GTM & architecture plan (remote caregiver + self-serve parent)

### 10.1 Product principles for trust

- **Data minimization:** collect only what’s needed for billing, activation, support/remote setup, and explicitly enabled safety features.
- **Transparent consent:** every new category (notifications, contacts, location, marketing) is opt-in with plain-language explanations.
- **Separation of roles:** caregivers have accounts; parent devices are linked with explicit consent and revocable access.
- **No resale of data:** no data is sold; limited, opt-in use for future safety app invitations only.
- **Right to revoke:** clear flows to unlink caregiver, disable location, and delete data.

---

### 10.2 Target data model (backend)

Even if not all used in Phase 1, design for:

- **User (Account)**  
  `{ userId, role: caregiver|self, email/phone, consentFlags, createdAt }`

- **Device**  
  `{ deviceId, platform, appVersion, pushToken, locale, createdAt }`

- **ProtectionLink** (caregiver ↔ parent’s device)  
  `{ linkId, caregiverUserId, deviceId, label, status, permissionsGranted, createdAt }`

- **Subscription**  
  `{ subscriptionId, ownerUserId, plan, status, providerRef, renewalAt, createdAt }`

- **Activation**  
  `{ code/token, subscriptionId, deviceId?, redeemedAt }`

This unlocks: robust support, multi-parent support, push notifications, future location, and cross-app invites without schema rewrites.

---

### 10.3 Phase 0 – UX-only wrapper (no backend dependency)

**Goal:** Validate onboarding UX and flows with current app; no network dependency.

- Add **SetupModeScreen** before onboarding:
  - “I’m setting this up for my parent”
  - “I’m setting this up for myself”
- Add **GetProtectedScreen**:
  - Primary: “Activate with link/code”
  - Primary: “Subscribe now” (stubbed for now)
  - Secondary: “Ask family to sponsor me” (share text only)
- After either path (mocked success), continue to existing `OnboardingFlow → LaunchGate → MainShell`.
- Locally record:
  - anonymized install id
  - chosen mode (caregiver vs self)
  - language and app version (for UX research)

No backend yet; this is just flow and copy.

---

### 10.4 Phase 1 – MVP GTM (web paywall + activation codes)

**Goal:** Remote caregiver can pay; parent can activate; app works end-to-end.

**Backend:**
- Hosted checkout (Stripe/Razorpay/Paddle) with:
  - plan (monthly/yearly)
  - optional promo/cheat code
- On success:
  - create `Subscription` (payer email/phone)
  - create `Activation` code/link
- API:
  - `POST /activate`  
    Request: `{ code, deviceId, parentPhoneOrName? }`  
    Response: `{ activated: true/false, status, expiresAt }`

**App:**
- `GetProtectedScreen` → “Activate with link/code”:
  - collect code (or handle deep link)
  - call `/activate`
  - on success, store `activationToken`/`expiresAt` in secure storage and mark `activated=true`.
- After activation: existing onboarding → LaunchGate → MainShell.

**Trust UX (must-have):**
- Add a **Privacy & Consent** step in onboarding:
  - clarify that data is used only for protection and never sold.
  - explicit opt-in checkbox: “Tell me about future safety apps we build” (off by default).

---

### 10.5 Phase 2 – User management + notifications (Firebase)

**Goal:** Proper account system for caregivers and robust device notifications.

**Backend / Identity:**
- Use **Firebase Authentication** (email/phone OTP) for caregivers.
- Parent devices can remain device-based initially; optionally allow them to create accounts later.

**Notifications:**
- Use **Firebase Cloud Messaging (FCM)**:
  - app registers `pushToken` with backend as part of `Device` record.
  - used for onboarding reminders, risk notifications (caregiver-aware where appropriate), and subscription updates.

**Linking upgrades:**
- Caregiver portal:
  - shows list of `ProtectionLinks` (e.g. “Mom’s phone – Activated”).
  - “Invite new parent device” → generates invite link (wraps activation code + caregiver identity).
- Parent app:
  - supports redeeming invite link, which creates a `ProtectionLink` (caregiverUserId + deviceId).

---

### 10.6 Phase 3 – Self-serve parent subscription + sponsorship

**Goal:** Parent can subscribe directly, or request sponsorship from family.

**Self-serve:**
- On `GetProtectedScreen`, “Subscribe now”:
  - opens hosted checkout with a flag that this is a **self** subscription.
  - after payment, backend:
    - creates `User` with role `self`
    - creates `Subscription` for this user
    - immediately associates current `deviceId` as activated (no code needed).

**Sponsorship flow:**
- Parent taps “Ask family to sponsor me”:
  - backend creates a **sponsorship link** referencing this `deviceId`.
  - link opens caregiver paywall; when paid, backend ties `Subscription` to that existing `deviceId`.

The app UI remains the same; only backend routing differs.

---

### 10.7 Phase 4 – Future premium: location for dementia/Parkinson’s

**Goal:** Add high-trust location features without turning into surveillance.

**Consent model:**
- Separate **Location Safety** section in settings:
  - clear explainer: “Used to help your family find you if you are lost or disoriented. You can turn this off any time.”
  - explicit toggles:
    - “Share last known location with my caregiver”
    - (Optional later) “Allow live location in emergencies”
- Parent can:
  - pause location
  - see recent “location access” events (who requested and when).

**Technical:**
- Use OS-compliant background location (persistent notification on Android, strict intervals).
- Backend stores:
  - location samples with strict retention (e.g. 24h or 7d) and access logs.
- Caregiver portal:
  - simple “Locate now” button; maybe later geofence alerts with explicit opt-in.

---

### 10.8 Phase 5 – Cross-app invitations (future apps)

**Goal:** Reuse trust and relationships for future safety apps without violating intent.

- Maintain a separate **marketing consent flag** per user:
  - e.g. `consentFlags: { futureAppsEmail: true/false }`.
- Use for:
  - low-frequency invitations (“We built a new safety app…”) clearly labeled as such.
  - one-click unsubscribe / in-portal toggle.
- Never:
  - sell or share data with third parties.
  - mix marketing with critical safety notifications.

This keeps Elder Shield’s mission and trust intact while allowing ethical future promotion.
